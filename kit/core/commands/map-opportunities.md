# /map-opportunities

## Goal

把盘点到的资产与痛点编译成 **agent 机会矩阵**:按「产品线 × 干系人视角」系统性扫描哪里值得上 agent,逐条产品线评估数据就绪度、痛点强度与改造量。产出 `agents/_portfolio/01-opportunity-matrix.md`,让「先做哪个 agent」变成有依据的排序题而不是拍脑袋。

## Required Inputs

- `agents/_portfolio/00-discovery.md`(前置产物,缺失则先执行 `/discover-business`)
- `business-agent/business-profile.yaml`
- `business-agent/core/templates/opportunity-matrix.template.md`

## Execution Rules

- **视角固定七类,不增不减**:客户 / 供应商 / 销售 / 运营 / 合规风控 / 工程 / 管理层。某视角对某产品线确实无机会时明确写「—」,不留空——空格意味着「没扫过」,「—」意味着「扫过没有」。
- **产品线来自 discovery,不凭空发明**:每条候选产品线必须能指回 discovery 中的系统、数据或痛点条目(用 `§章节号` 引用);discovery 里没有依据的想法登记到「盘点缺口」,回补 discovery 后再入矩阵。
- 每条产品线逐项登记:服务对象、现状痛点、数据基础(引用 discovery 条目)、核心能力表(能力 | 用户示例问题 | 所需工具)、改造量(低 / 中 / 高)。
- **三维评分口径统一**:
  - 数据就绪度:高=已有 API 或结构化报表可直接查;中=数据存在但要新开接口 / 清洗;低=数据缺失或散落在人脑与聊天记录里。
  - 痛点强度:用 discovery 的「频次 × 耗时 × 涉及角色」折算,高频高耗时高。
  - 改造量:低=纯只读查询包一层工具;中=需要新开 1~2 个后端接口或聚合层;高=需要新建数据管道 / 改造核心系统。
- **只读优先原则在本阶段就体现**:能力表中把只读查询类能力与写操作类能力分开标注(`read` / `write`),为 roadmap 的波次排布做好输入。
- 评分与用户逐条过一遍并记录分歧;用户不认可的评分按用户口径修正,但保留「AI 初评 vs 修正后」两列备查。

## Exit Criteria

- [ ] 全景矩阵覆盖全部候选产品线 × 七视角,每格要么有机会描述要么是「—」,无空格。
- [ ] 每条产品线的痛点与数据基础都有 discovery 章节引用,无凭空条目。
- [ ] 三维评分逐线完成且口径一致;能力表区分了 read / write。
- [ ] 评分与用户核对过,分歧有记录;综合建议(优先 / 观察 / 暂缓)已给出。
- [ ] 全文无真实密钥、内部域名、真实客户数据。
