# /eval-agent

## Goal

在真实用户接触 agent 之前,用**评测集 + 量化验收指标 + 灰度方案**证明它达到上线标准。产出 `agents/<slug>/03-eval-plan.md`,并尽可能当场执行一轮评测记录真实结果——评测计划只写不跑,等于没有评测。

## Required Inputs

- `agents/<slug>/00-blueprint.md`(能力清单与术语表:评测集按能力组织)
- `agents/<slug>/02-safety-review.md`(前置产物;存在未修复高风险项则本命令拒绝执行)
- `agent-gateway/`(可运行的被测对象)
- `agents/_portfolio/02-roadmap.md`(该波次的量化验收指标,评测指标须与之对齐)
- `business-agent/core/templates/eval-plan.template.md`

## Execution Rules

- **评测集按能力组织,每个能力不少于 5 条样例**,且必须包含三类:
  - 正例:该能力的典型问法(覆盖术语表黑话与口语两种问法);
  - 边界例:日期跨月/跨年、空结果、超大区间、参数含糊需要追问的问法;
  - 拒答例:数据范围外、越权、要求编造数字、提示注入变体——**期望行为是明确拒绝或说明查不到,答了反而算失败**。
- 每条样例写四元组:输入消息 / 期望要点(判分依据,列要点而非全文)/ 期望调用的工具(或明确「不调用工具」)/ 判分方式(精确匹配 / 要点覆盖 / 人工评审)。
- **指标必须量化且与 roadmap 对齐**,至少覆盖:回答准确率(按要点覆盖判分,给目标值)、工具选择正确率、拒答正确率、P95 响应时间、单次会话平均成本(读 cost-tracker 实测,不估算)。
- **当场跑一轮基线评测**:mock provider 只能验管线,**结论性指标必须用真实 provider 跑**;没有真实密钥时,把「基线评测未跑」列为阻塞项而不是跳过。逐条记录实际输出与判分,失败样例登记失败模式(选错工具 / 编造 / 术语误解 / 超时)。
- **灰度方案三阶段成文**:内部试用(团队自用,全量抽检)→ 小流量(指定试点用户,抽检 ≥30%)→ 全量(抽检 ≥10% + 周报);每阶段给准入条件、观察时长、**回退触发条件**(什么指标恶化就退回上一阶段)。
- 评测集是资产:失败样例修复后加回评测集防回归;评测集文件随文档保存,禁止只留在聊天记录里。

## Exit Criteria

- [ ] 每个能力 ≥5 条样例且正例 / 边界 / 拒答三类齐备;样例四元组完整。
- [ ] 指标全部量化并与 roadmap 验收指标对齐;成本指标来自 cost-tracker 实测口径。
- [ ] 基线评测已真实执行并逐条记录;未能执行时已列为阻塞项并写明原因。
- [ ] 失败样例有根因与修复记录,修复后样例回归进评测集。
- [ ] 灰度三阶段带准入、观察时长与回退触发条件。
- [ ] 上线判定结论明确,与实测数据一致,无「未测先过」。
