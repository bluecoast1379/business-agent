# 护栏(Guardrails)

护栏不是「让 agent 更听话」,而是**假定它一定会出错 / 被注入,把爆炸半径钉死**。四件套:数据隔离三层、写操作人工确认、预算双层、fail-fast 配置。

## 1. 数据隔离三层保障

以 Brewline 为例:咖啡馆客户「云雾庄园」的采购经理通过 agent 查数据,绝不能看到别家咖啡馆的订单。三层设防:

| 层 | 措施 | 性质 |
|---|---|---|
| 提示词层 | 系统提示词声明「你只能查询当前租户(云雾庄园)的数据,拒绝任何跨租户请求」 | 纵深防御,**可被注入绕过** |
| 工具层 | `withScope` 把 `tenant_id` 强制注入每次工具调用,模型传入的同名参数被覆盖 | 纵深防御,防模型失误与多数注入 |
| 后端层 | 后端按请求携带的 token 校验数据归属,越权直接 403 | **唯一真正的安全边界** |

必须刻在墙上的一句话:**前两层只是纵深,只有后端层是安全边界。** 提示词会被「忽略之前的指令」击穿;工具层拦得住模型传错参,拦不住网关代码本身的 bug 或凭证泄露。评审时的判定标准也据此定:后端层缺失 → 安全审查整体不通过;前两层缺失 → 高优先级修复项。

反例:团队看到工具层注入做得很好,认为后端可以开一个「信任网关」的免鉴权端点省一次校验。这等于把三层拆成一层——网关一旦被攻破(或凭证泄露),全库裸奔。

## 2. 写操作人工确认(带外审批确认门)

所有对业务系统产生副作用的工具(建单、改状态、发消息)必须包确认门:

1. **第一段**:模型调用写工具 → 网关**不执行**,把待执行动作登记进确认中心,返回「待执行摘要 + confirmationId」。摘要用人话写清副作用:「将为发票 INV-2024-0731 创建贷记单,金额 320 元,原因:运输破损」。
2. **人工带外审批**:审批信号**不经过 agent 循环**——操作员通过鉴权后的管理端点(如 `POST /confirmations/:id/approve`)或 REPL 人工命令批准。模型拿着 confirmationId 也无法批准自己。
3. **第二段**:审批通过后,凭 confirmationId 二次调用才真执行;未审批的二次调用一律拒绝;id 短时效、一次性,过期作废。

要点:确认对象是**人**,且确认信号必须走模型触达不到的通道——任何把确认凭证带内返回给模型的实现,模型都能在同一循环里自我确认,门形同虚设;确认条目绑定第一段的完整参数,审批后参数不可再变(防「摘要说 320,执行时改 3200」);每个写路径都要 grep 核对没有绕过确认门的直调入口(事故清单第 9 项)。

反例一:确认实现成「模型再调一次同名工具就算确认」。注入话术只需让模型连调两次。
反例二:首调把确认 token 直接返回进对话,「凭 token 再调即执行」——token 在模型手里,等于没有门。

## 3. 预算双层

- **单次会话上限 `maxBudgetUsd`**:agent 循环每轮把 usage 折算成本累加,超限即停止本次回答并明确告知(防单会话死循环烧钱)。
- **月度总额**:cost-tracker 汇总所有 agent 的月成本,`isOverBudget` 命中后拒绝新请求、只保留 `/health`;70% 预警线提前通知(见 `/operate-agent` 三道线)。

两层缺一不可:只有单次上限,挡不住高频小额;只有月度总额,一次失控会话就能吃掉半月预算。另一个关键是**接线验证**——cost-tracker 写好了不等于生效,冒烟测试必须断言一次对话后成本读数 > 0(事故清单第 3 项);预算判断读 tracker 实时读数,不读配置常量。

## 4. fail-fast 配置与禁止默认凭证

配置层两条铁律:

1. **必填缺失就崩,启动即崩优于运行时怪象。** `LLM_PROVIDER`、`GATEWAY_AUTH_TOKEN` 之类必填项缺失时直接 throw 并写明缺哪个键,绝不静默降级。「服务起来了但 agent 一直答错」比「起不来」贵得多。
2. **任何配置项不得有可用的密钥 / 域名默认值。** `process.env.LLM_API_KEY || "sk-xxx"`、`BACKEND_URL || "http://10.x.x.x"` 都是事故清单第 1 项的标准姿势;默认值随代码进 git、进开源、进转发。demo 便利用 `LLM_PROVIDER=mock` 提供,不用真实凭证提供。

配套纪律:`.env` 与 `node_modules/` 必须在 `.gitignore`(事故清单第 2 项);密钥一旦进过 git 历史,处置动作是**轮换**,不是删文件。

## 5. 护栏与事故清单的对照速查

| 护栏 | 防的是什么 | 对应事故清单项 |
|---|---|---|
| 数据隔离三层 | 越权查询、提示注入拿他租户数据 | 6 |
| 两段式确认门 | 幻觉/注入直接落库发消息 | 9 |
| 预算双层 + 接线验证 | 会话死循环烧钱、月底账单失控 | 3 |
| fail-fast 配置 | 缺配置静默降级成运行时怪象 | 1 |
| 禁止默认凭证 + .gitignore | 密钥随代码泄露 | 1、2 |
| 端点强制 Bearer token | 预算被白嫖、工具面被探测 | 5 |

## 6. 护栏的验收方式

护栏写在蓝图里只是承诺,验收靠三样:

- **代码证据**:withScope / confirm-gate / fail-fast 的具体代码位置(`/harden-agent` 留证)。
- **实测证据**:越权请求被拒、无 token 401、注入样例未触发写工具、成本读数非 0。
- **持续证据**:拒答类评测样例常驻评测集(`/eval-agent`),每次改提示词都回归——护栏最常见的死法不是没建,而是某次「优化提示词」悄悄删掉了安全规则段。
