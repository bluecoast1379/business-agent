# 分层诊断引擎(Layered Diagnosis)

对账、稽核、数据体检这类「找出哪里不对」的任务,不要把原始数据丢给 LLM 让它「看看有什么问题」——又贵又不稳定。正确姿势是把诊断拆成**三层流水线**,让确定性代码干确定性的活,LLM 只在需要理解与归因的层面出场。

## 三层结构

以 Brewline 的月度对账为例:每月要核对「订单 → 出库 → 发票 → 回款」四个环节的数据是否闭环。

### 第 1 层:完整性检查(纯代码,零 LLM)

**问的是「该有的都在吗」。** 用确定性规则扫描:

- 每张已出库订单是否都有对应发票?(孤儿订单)
- 每张发票是否都能指回订单?(孤儿发票)
- 关键字段是否为空 / 格式非法?(缺税号的发票、金额为负的订单)
- 数量是否对得上?(本月订单 412 单 vs 发票 407 张,差 5)

产出:结构化差异清单 `[{type: "missing_invoice", order_id: "SO-1029", ...}]`。这一层跑得快、可重复、结果可信,是后两层的输入。

### 第 2 层:规则一致性校验(代码为主,规则外置)

**问的是「在的这些,值对吗」。** 对第 1 层过关的数据,逐条跑业务规则:

- 发票金额 = 订单金额 × (1 - 合同折扣)?
- 账期计算是否符合客户等级(金牌客户 60 天、普通 30 天)?
- 烘焙批次的出库量 ≤ 该批次入库量?

规则以配置 / 规则表形式外置(不硬编码在扫描器里),每条规则命中时输出「规则 ID + 实际值 vs 期望值 + 涉及单据」。产出:违规清单,按规则聚类。

### 第 3 层:人工干预识别(LLM 出场)

**问的是「剩下这些解释不了的,是什么情况」。** 前两层解释不掉的差异,多半是脏数据与人工干预:线下改过的单、手工冲账、补录错月份、客服私下答应的折让。这一层把**差异条目 + 相关单据摘要**(不是全量数据)交给 LLM:

- 归因假设:「发票 INV-0311 金额比规则值少 320,与同客户当月一张贷记单金额吻合,疑似运输破损折让」;
- 分类打标:脏数据 / 人工调整 / 规则未覆盖的新业务 / 真异常;
- 生成给人看的核对建议,**最终裁定永远留给人**。

## 为什么必须分层

- **成本**:第 1、2 层过滤掉 95% 以上的数据量,LLM 只看疑难杂症,token 支出降一个数量级(参见 `cost-model.md`)。
- **可信度**:「差 5 张发票」由代码数出来,永远数得一样;LLM 数数是出了名的不可靠。
- **可回归**:规则层每条规则可单测;LLM 层的归因质量用评测集抽查。混在一起则什么都测不了。

## 反例

- **反例 1:把两千行对账明细直接贴给模型「找异常」。** 模型挑几条显眼的,漏掉大半,还会把正常的批量折扣当异常报;每跑一次结果不同,没人敢据此做账。
- **反例 2:规则硬编码在提示词里。** 「金牌客户账期 60 天」写进系统提示词,财务改成 45 天后没人记得改提示词,agent 连续一个月误报。守卫:规则外置,提示词只教模型怎么读规则命中结果。
- **反例 3:LLM 归因结果直接自动冲账。** 第 3 层输出是**假设**,不是裁定;自动执行等于让幻觉直接改账。守卫:第 3 层只产出建议清单,写操作走确认门。

## 实现落点

在 scaffold 中,分层诊断适合实现为 patrol agent 的任务体:第 1、2 层是普通函数(可独立单测),第 3 层通过临时 agent 实例调用 LLM,任务提示词里给「编号步骤 + 输出格式」(见 `patrol-agents.md`)。阈值(如「差异超过 N 条才升级告警」)外置到环境变量。

```js
// pipeline sketch: deterministic first, LLM last
async function monthlyReconciliation(range) {
  const gaps = integrityScan(range);        // layer 1: pure code
  const violations = ruleCheck(range);      // layer 2: rules from config
  const unexplained = residual(gaps, violations);
  if (unexplained.length === 0) return report({ gaps, violations });
  const attribution = await patrolAgent.prompt(
    buildAttributionTask(unexplained, range)  // layer 3: summaries only
  );
  return report({ gaps, violations, attribution }); // suggestions, not verdicts
}
```

## 落地检查清单

- [ ] 第 1、2 层无 LLM 调用,可离线单测,同输入必同输出。
- [ ] 规则外置(配置/规则表),每条有 ID,命中输出「实际 vs 期望 + 单据号」。
- [ ] 第 3 层只接收差异条目与单据摘要,不接收全量数据。
- [ ] LLM 输出定位为「归因假设 + 建议」,任何账务动作走人工确认门。
- [ ] 升级告警阈值来自环境变量;诊断报告带数据区间与运行时间戳。
