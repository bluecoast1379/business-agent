# 成本模型(Cost Model)

LLM 成本是业务 agent 唯一随用量线性增长的支出,必须在 roadmap 阶段就建模、在运行时实测、在运营期复核。本篇给估算方法、分档策略与省 token 手段;**不给任何具体价格**——模型单价变动频繁,一切以 provider 官方价目为准,并记录查询日期。

## 1. 估算方法:场景 × 日均次数 × token 量 × 档位

对每个使用场景单独建一行账:

```
月成本(场景) = 日均调用次数 × 30
             × [ 单次输入 tokens × 输入单价 + 单次输出 tokens × 输出单价 ]
```

估 token 量的实操口径:

- **输入 = 系统提示词 + 会话历史 + 工具返回**。工具返回常是大头——这就是 summary/raw 双形态直接省钱的原因(见 `tool-design.md`)。
- 中文粗估 1 字 ≈ 1~2 tokens,英文 1 词 ≈ 1.3 tokens;精确值用 provider 的 token 计数接口实测 3~5 条典型对话取均值。
- 多轮工具调用要按**轮次累加**:一次「查订单再查发票」的回答,输入要算两轮(每轮都重复带上下文)。
- 巡检场景按「每次运行的完整轮次 × 每日运行次数」估,别按一次调用估。

以 Brewline 为例,一张最小可用的估算表长这样(数字均为示例口径,单价留待查询):

| 场景 | 日均次数 | 轮次 | 单次输入 tokens(估) | 单次输出 tokens(估) | 档位 | 月成本算式 |
|---|---|---|---|---|---|---|
| 运营日常查询 | 40 | 1~2 | 3,000 | 400 | 默认档 | 40×30×(3000×In + 400×Out) |
| 逾期发票日巡 | 1 | 5 | 12,000(逐轮累积) | 1,500 | 默认档 | 1×30×(12000×In + 1500×Out) |
| 对账归因(月度) | 1/30 | 3 | 20,000 | 3,000 | 复杂档 | 1×(20000×In2 + 3000×Out2) |

其中 In/Out 为默认档输入/输出单价、In2/Out2 为复杂档单价,查询日期与价目页记录在 roadmap。分行列出才能看清「原来巡检比问答贵」这类反直觉结论,也才知道优化该先动哪一行。

## 2. 模型分档:便宜默认档 + 复杂任务档

- **默认档**:便宜快的模型,承接绝大多数流量——查数、汇总、格式化、按模板输出。巡检任务因为提示词高度结构化,默认档通常足够。
- **复杂任务档**:强模型,只在明确条件下升档:多步推理归因(如分层诊断的第 3 层)、长文档理解、生成面向客户的正式文案。
- 升档条件写成**路由规则**落在蓝图 §7(如「消息含附件文档 → 复杂档」「patrol 第 3 层归因 → 复杂档」),不靠临场感觉。
- 分档收益经验量级:八成以上流量走默认档时,总成本约为全量强模型的 1/5~1/3(具体取决于两档价差,按当前价目自算)。

配置上两档模型名都来自环境变量(`LLM_MODEL` / `LLM_COMPLEX_MODEL`),换代升级不改代码;**每季度复核一次型号与价目**(`/operate-agent` SOP 4.3)——模型市场半年一换代,去年的「贵档」可能已比今年的「便宜档」又贵又差。

## 3. 省 token 的三板斧

1. **摘要化工具返回**(最大头):summary 默认、raw 限行、hint 引导下钻;把「表格转述给模型」改成「聚合结果给模型」。
2. **会话历史裁剪**:超过 N 轮的旧对话做摘要压缩或直接截断;巡检用临时实例天然零历史。
3. **提示词缓存**:系统提示词 + 工具定义在多数 provider 支持前缀缓存,把稳定部分(角色、术语表、工具 schema)放前面、时变部分(日期、会话)放后面,命中缓存的输入按折扣价计。确认所用 provider 的缓存计价规则后再依赖此项。

## 4. 运行时实测与对账

估算只用于立项;上线后一切以 **cost-tracker 实测**为准:

- agent 每轮把 usage(输入/输出 tokens)按单价表折算上报,单价表可 env 覆盖(价目调整时改配置);
- 上报时带 agent 名与场景标签,月报才能按场景归因「钱花在哪一行」,与 §1 估算表逐行对照校准;
- `/status` 暴露月累计,与预算三道线(70% 预警 / 100% 熔断)联动;
- 每月人工对账一次:tracker 读数 vs provider 账单,偏差超 10% 要查(常见原因:某调用点没上报 usage——事故清单第 3 项的运行期变体);
- 评测也是成本:`/eval-agent` 基线评测与每次回归的调用同样计入 tracker,预算里给评测留出固定份额(经验值 5%~10%),避免「为省钱不敢跑回归」。

## 反例

- **反例 1:roadmap 里写死单价。** 三个月后价目调整,整份预算表作废还没人发现,继续拿旧数汇报。守卫:表里只留算式与「查询日期 + 价目页」,数字每季度重算。
- **反例 2:按「一次调用」估巡检成本。** 实际每次巡检 5 轮工具调用、输入逐轮累积,月账单是估算的 6 倍。守卫:按轮次累加估,上线第一周就对照实测校准。
- **反例 3:为省钱全量切最便宜模型。** 归因类任务质量崩了,人工复核工时把省下的 API 钱翻倍赔进去。守卫:分档路由,用评测集验证降档不掉质量再切。
