# 业务 agent 网关十大事故清单

> 供 `/harden-agent` 逐项对照。每项按「症状 → 根因 → 守卫」组织;审查时对每项给出结论(通过 / 不通过 / 不适用)与**可复核的证据**(文件路径、grep 命令、测试输出),不接受「应该没问题」。

## 1. 源码内兜底密钥 / 内部域名

- **症状**:环境变量缺失时服务照常启动并能调通 LLM 或后端;仓库开源或转发后,密钥与内部地址随源码泄露。
- **根因**:开发图省事写 `process.env.LLM_API_KEY || "sk-xxxx"` 或把内网网关地址当默认值;评审只验证功能、不看默认值分支。
- **守卫**:配置层 fail-fast——必填环境变量缺失直接抛错退出,任何配置项不得有可用的密钥 / 域名默认值;发布前跑脱敏扫描(`npm run check` 内的 check-sanitized);代码评审专项 grep 所有 `||`、`??` 默认值与形如密钥的字面量。

## 2. `.gitignore` 不含 `.env` / `local` / `node_modules`

- **症状**:`.env`、`local/state.json`、同目录 lock/tmp/bak 或依赖目录被提交,凭证、业务状态或审批元数据进入 git 历史;之后即使删除文件,历史提交里仍能翻出。
- **根因**:实例化 scaffold 时没有带出 `.gitignore`(复制工具跳过隐藏文件),或漏了相对 cwd 生成的 `scaffold/local/` 运行时产物。
- **守卫**:scaffold 自带 `.gitignore`(至少含 `.env`、`/local/`、`node_modules/`、`*.log`);初始化工具幂等确保目标工作区 `.gitignore` 收录私有路径和 `business-agent/scaffold/local/` 防御性规则;smoke 用真实 `git check-ignore` 验证 state/lock/tmp/bak;一旦发现密钥或业务状态已入历史,**必须轮换密钥并评估状态暴露**,只删文件不算处置。

## 3. 月度成本追踪定义了未接线

- **症状**:`/status` 端点或成本报表永远显示 0;预算护栏形同虚设,直到月底账单异常才被发现。
- **根因**:cost-tracker 模块写好了,但 agent 的 LLM 调用循环没有调用 `trackUsage`——「模块存在」被误当成「功能生效」。
- **守卫**:冒烟测试必须断言一次真实对话后 `costUsd > 0`(证明已接线);评审把「每个 LLM 调用点都上报 usage」列为专项检查项;预算超限判断读的是 tracker 的实时读数而非配置常量。

## 4. 系统提示词日期在模块加载时求值

- **症状**:长驻进程运行数天后,agent 仍认为「今天」是进程启动日,所有相对日期(昨天、本月、上周)系统性偏移。
- **根因**:`const SYSTEM_PROMPT = \`今天是 ${today}\`` 之类的模板在模块顶层求值一次,进程不重启就不更新。
- **守卫**:系统提示词用**函数**在每次请求时组装,日期、余额等时变信息一律运行时求值;评测集包含「今天是几号 / 本月订单」类用例,并要求在长驻(非刚启动)环境复测一次。

## 5. 聊天端点无鉴权 / 无限流

- **症状**:网关端口暴露后,任何人都能白嫖你的 LLM 预算、探测你注册了哪些内部工具,甚至驱动写操作。
- **根因**:demo 阶段图方便不加 token,上线时忘了补;或以为「内网部署」等于安全边界。
- **守卫**:除 `/health` 外所有端点强制 `Authorization: Bearer <GATEWAY_AUTH_TOKEN>`,token 必填、无默认值;冒烟测试断言无 token 请求返回 401;对话端点叠加会话级预算上限,天然构成粗粒度限流。

## 6. 默认以最高权限角色调用后端

- **症状**:agent 能查到(甚至改到)当前服务对象无权访问的数据;一次成功的提示注入即等于全库越权。
- **根因**:网关持有管理员级后端凭证,工具层不区分「替谁调用」,把权限判断完全寄托在提示词上。
- **守卫**:为 agent 单独申请**最小权限**后端角色(只授予工具清单实际需要的读写范围);工具层 scope 绑定强制注入租户 / 门店标识,外部传入的同名参数被覆盖;后端按请求方 token 校验数据归属——**只有后端层是真正的安全边界**。

## 7. 知识文件存在但从未注入提示词

- **症状**:术语表、业务规则文档写得很全,agent 回答仍然不懂行话;排查发现提示词组装代码根本没有读它。
- **根因**:知识文件与提示词组装逻辑脱节;或文件改名 / 移动后引用失效,代码静默跳过不报错。
- **守卫**:提示词组装时**运行时读取**知识文件并拼入,文件读不到直接抛错而不是静默降级;评测集包含术语表词条问答(答不出即视为注入失效);知识文件路径进配置而非散落硬编码。

## 8. 入站 webhook 未验签 / 未解密即信任

- **症状**:任何拿到 webhook URL 的人都能伪造「用户消息」驱动 agent 执行工具,相当于绕过全部鉴权。
- **根因**:只实现了「收到消息 → 处理」,跳过签名校验;或验签代码存在但失败时只打日志、继续处理。
- **守卫**:入站 webhook 必须做 HMAC(或渠道方规定的)验签,secret 来自环境变量;**验签失败必须以 4xx 拒绝并告警**,绝不降级放行;渗透用例包含伪造签名与重放请求;消息体需要解密的渠道,解密失败同样拒绝。

## 9. 写操作无人工确认

- **症状**:模型幻觉或提示注入直接触发落库、发消息、改单据,损失不可逆且难以追责。
- **根因**:写工具与读工具被同等注册、同等调用;过度信任「模型不会乱来」。
- **守卫**:所有 write 工具必须包带外审批确认门——首次调用只登记待办并返回「待执行摘要 + confirmationId」,**人**经模型触达不到的通道(鉴权管理端点 / REPL 人工命令)审批后,凭短时效一次性 id 二次调用才真执行,未审批的二次调用一律拒绝;注意「把确认令牌带内返回给模型」的两段式是假门,模型可自我确认;工具契约中 `write: true` 必须伴随 `confirm: human`;审查时 grep 全部写路径确认无绕行入口。

## 10. 工具返回原始大 JSON 撑爆上下文

- **症状**:一次列表查询返回数千行 JSON,token 成本暴涨、响应变慢,模型注意力被无关字段稀释,回答质量不升反降。
- **根因**:把后端响应原样透传给 LLM;工具只有一种「全量」形态。
- **守卫**:工具默认返回 **summary 形态**(裁剪字段、截断行数、给聚合统计),raw 形态需显式请求且仍限制行数;返回体携带 `hint` 字段告诉模型「数据被截断,可用 X 参数下钻」;评审对每个查询工具核对最大返回体积。
