## 3. 产品线详述(每线一节)

### 3.1 运营问数助手

- 服务对象:销售、运营、财务等内部员工(discovery §6)。
- 现状痛点(引用 discovery §):§4 门店账单答疑日均约 15 次×10 分钟;各角色临时问数都要找财务或自己导表。
- 数据基础(引用 discovery §):§2 Brewline ERP(订单 / 发票 / 账期 / 订阅)REST API 齐全。
- 核心能力表:

| 能力 | 用户示例问题 | 所需工具(读/写) |
|---|---|---|
| 订单汇总查询 | 「青柠巷店这两个月订了多少豆?」 | get_order_summary(read) |
| 发票与账期查询 | 「INV-2026-0612 现在什么状态?」 | get_invoice_detail / get_customer_billing_profile(read) |
| 未回款清单 | 「哪些门店还有没回款的发票?」 | get_unpaid_invoices(read) |

- 改造量评估与理由:低——纯只读查询包一层工具,复用现有 API。
- 风险与前提:提示词须注入账期术语表,否则口径答错(见蓝图 §2.3)。

### 3.2 对账诊断

- 服务对象:财务负责人、财务专员(discovery §6)。
- 现状痛点(引用 discovery §):§4 周对账 3 小时/次、逾期跟催 1 小时×2 次/周,差异定位靠肉眼。
- 数据基础(引用 discovery §):§2 ERP(发票 / 订单 / 回款镜像)、§3.3 PayFlow 回款事件字段完备。
- 核心能力表:

| 能力 | 用户示例问题 | 所需工具(读/写) |
|---|---|---|
| 三方核对 | 「上周的回款和发票对得上吗?」 | get_payment_records / get_unpaid_invoices(read) |
| 差异分层诊断 | 「6 月差异帮我按原因分个类」 | run_reconciliation_diagnosis(read) |
| 逾期画像 | 「逾期 30 天以上的门店有哪些?」 | get_unpaid_invoices(read) |
| 贷记单创建 | 「给 INV-2026-0587 开 240 元贷记单」 | create_credit_note(write,人工确认) |
| 催款提醒 | 「给逾期两周的门店发提醒」 | send_payment_reminder(write,人工确认) |

- 改造量评估与理由:中——只读部分零改造;诊断需一个本地规则脚本(完整性 → 一致性 → 人工干预清单三层)。
- 风险与前提:写能力(贷记 / 催款)必须人工确认并后置到写操作波次。

### 3.3 客户健康度

- 服务对象:销售总监、运营负责人(discovery §6)。
- 现状痛点(引用 discovery §):§4 门店流失后知后觉,月复盘才发现降档 / 停订。
- 数据基础(引用 discovery §):§3.2 门店订阅档位标签覆盖 100%;§3.3 订单事件流可算下单频率变化。
- 核心能力表:

| 能力 | 用户示例问题 | 所需工具(读/写) |
|---|---|---|
| 健康度评分 | 「哪些门店最近三个月订货量下滑超三成?」 | 订单趋势查询(read,需聚合接口) |
| 异动预警巡检 | (定时)「本周订阅降档 / 停订清单」 | 订阅异动查询(read) |

- 改造量评估与理由:中——需要 ERP 新开 1 个订单趋势聚合接口。
- 风险与前提:评分口径需销售与运营先对齐,否则预警没人认。

### 3.4 供应链监控

- 服务对象:采购经理(discovery §6)。
- 现状痛点(引用 discovery §):§4 交期盯梢每工作日 20 分钟,延迟发现晚影响烘焙排产。
- 数据基础(引用 discovery §):§2 Roastery OS 只读 API(批次 / 库存 / 在途);§3.2 供应商交期达成率标签。
- 核心能力表:

| 能力 | 用户示例问题 | 所需工具(读/写) |
|---|---|---|
| 在途与库存查询 | 「水洗耶加豆仓还够撑几周?」 | 库存 / 在途查询(read) |
| 交期异常巡检 | (定时)「发运延迟超 3 天的采购单」 | 采购单状态查询(read) |
| 供应商表现汇总 | 「近两季各庄园交期达成率排一下」 | 供应商表现查询(read) |

- 改造量评估与理由:中——Roastery OS 接入是新集成,但只读、接口现成。
- 风险与前提:交期达成率标签人工维护滞后(§3.2),需先补数据 SOP。

### 3.5 运营报表

- 服务对象:总经理、运营负责人(discovery §6)。
- 现状痛点(引用 discovery §):§3.1 经营周报人工拼表 2 小时/周,口径偶有出入。
- 数据基础(引用 discovery §):§2 ERP 数据齐;§3.1 周报字段口径已稳定。
- 核心能力表:

| 能力 | 用户示例问题 | 所需工具(读/写) |
|---|---|---|
| 周报自动生成 | (定时)每周一生成经营周报并推送 | 订单 / 回款 / 订阅汇总查询(read) |
| 口径追问 | 「这期周报的回款口径含贷记吗?」 | 术语表 + 汇总查询(read) |

- 改造量评估与理由:中——查询现成,需把人工口径固化为提示词与模板。
- 风险与前提:先与财务锁定口径,否则自动周报会放大口径分歧。

### 3.6 客户自助助手

- 服务对象:咖啡馆门店(外部客户,discovery §1)。
- 现状痛点(引用 discovery §):§4 门店账单答疑日均 15 次;§5 客户门户无对话能力。
- 数据基础(引用 discovery §):§2 数据在 ERP 齐备,但客户门户无 API、需新开门店级接口。
- 核心能力表:

| 能力 | 用户示例问题 | 所需工具(读/写) |
|---|---|---|
| 门店自查订单 / 配送 | 「我上周的豆子发货了吗?」 | 门店级订单查询(read,门店 scope) |
| 门店自查发票 / 账期 | 「我这期账单为什么多了 300?」 | 门店级发票查询(read,门店 scope) |

- 改造量评估与理由:高——对外暴露,需门店级 scope token、限流、独立渗透测试与新开接口。
- 风险与前提:必须在内部 agent 跑顺、护栏经受住检验之后再对外。
