# 方法论:从零规划业务 AI Agent

本文是 10 个阶段命令的叙事版:为什么是这个顺序、每一步在回答什么问题、产物长什么样。命令的精确契约以 `kit/core/commands/<id>.md` 为准;本文帮助你理解**为什么**。全文以虚构公司 **Brewline**(精品咖啡豆 B2B 供应商,客户是咖啡馆)为例,数据全部合成。

## 为什么先规划,再写代码

业务 agent 项目最常见的死法不是模型不行,而是:

1. **跳过盘点直接写代码**——agent 上线后才发现关键数据在另一个系统里、没有 API、或者拿不到权限;
2. **只从工程视角选题**——做了一个技术上漂亮但没人用的 demo,真正高频的痛点(比如每周花一下午的人工对账)没人碰;
3. **写操作裸奔**——agent 直接调用会改数的接口,一次幻觉就是一次生产事故;
4. **成本与运营失控**——上线时没人算过每天会烧多少钱,也没人负责看巡检结果。

所以本方法论把流程压成一条硬顺序:**先盘资产,再选机会,再排波次,然后才允许为单个 agent 画蓝图、编工具契约;代码生成设有实现闸门;上线前必须过安全审查与评测;上线后必须有巡检与成本模型。** 随时用 `/agent-status` 查看每个 agent 走到哪一步、缺什么产物。

```
discover-business ─→ map-opportunities ─→ plan-roadmap        (组合层:一次)
                                             │
                    ┌────────────────────────┘
                    ▼
design-agent ─→ define-tools ─→ [闸门] scaffold-gateway
                    │                        │
                    ▼                        ▼
              harden-agent ─→ eval-agent ─→ operate-agent      (单 agent 层:每个 agent 一轮)
```

## 第 1 步:业务资产盘点(`/discover-business`)

回答的问题:**我们手里到底有什么?**

以访谈方式盘点五类资产,产出 `agents/_portfolio/00-discovery.md`:

| 类别 | 登记内容 |
| --- | --- |
| 现有系统 / API | 名称、用途、读 / 写、认证方式、环境变量**键名** |
| 数据资产 | 报表、标签、事件流、日志——在哪、多新、谁维护 |
| 人工流程痛点 | 做什么、**频次 × 单次耗时**、出错代价 |
| 通知渠道 | IM 群机器人、邮件、站内信——出站与入站能力 |
| 干系人角色 | 谁提问、谁审批、谁背锅 |

两条铁律:

- **只登记「键名与用途」,严禁把密钥值写进文档。** 正确写法:`Brewline ERP,Bearer token,键名 BACKEND_URL / BACKEND_API_KEY`;错误写法:把 token 值贴进表格。
- **缺答案不猜。** 拿不准的条目列成 open questions 写进 `INITIALIZATION_QUESTIONS.md`,后续命令看到未闭环条目会提醒你。

Brewline 小例:盘点发现 ERP(订单 / 发票 / 账期)有完整 REST API,烘焙排产系统(批次 / 杯测分)只读可取,支付服务商 PayFlow 有回款流水 API 和带 HMAC 验签的入站 webhook——而财务每周一花 3 小时人工对账。这一行盘点直接决定了后面「对账诊断」会成为第一波产品线。

## 第 2 步:机会矩阵(`/map-opportunities`)

回答的问题:**这些资产能长出哪些 agent 产品线?各自值不值得做?**

方法论核心是一张「**产品线 × 干系人视角**」矩阵。视角固定七类:**客户 / 供应商 / 销售 / 运营 / 合规风控 / 工程 / 管理层**。固定七视角的目的就是防止只从工程视角选题——每条候选产品线都要回答「合规风控怎么看」「管理层能拿到什么」,答不上来的候选自然沉底。

每条候选产品线登记五件事:

1. 服务对象(引用 discovery 的角色条目);
2. 现状痛点(引用 discovery 的痛点条目,带频次 × 耗时);
3. 数据基础(引用 discovery 的系统 / 数据条目——**没有数据基础的机会是幻觉**);
4. 核心能力表:`能力 | 示例问题 / 动作 | 所需工具`;
5. 改造量:低 / 中 / 高(现有 API 够不够、要不要后端配合开新接口)。

Brewline 的矩阵收敛为六条产品线:运营问数助手、对账诊断、客户健康度、供应链监控、运营报表、客户自助助手(完整样例见 `examples/brewline/_portfolio/01-opportunity-matrix.md`)。

## 第 3 步:波次规划(`/plan-roadmap`)

回答的问题:**先做哪个?每一波怎么算账、怎么验收?**

排序公式:**数据就绪度 × 痛点强度 × 风险**。其中风险维度有一条不可协商的规则:

> **只读先行,写操作后置。** 第一波只做读查询与诊断;任何会改数、发消息给客户的能力,排到团队已经跑顺只读波次、且写操作确认闸(confirm-gate)已验证之后。

建议 3~4 波,每一波给三张表:

- **优先级表**:产品线、依赖、改造量、负责人;
- **验收指标**:必须量化(如「财务周对账人工耗时从 3 小时降到 30 分钟」「查询命中率 ≥ 95%」),不接受「体验更好」;
- **成本估算表**:`场景 × 日均调用次数 × 单次 token 量 × 模型档位单价`。kit 只给估算**方法**,不给具体价格——模型价格变动频繁,规划时查询 provider 当前定价页,并在运营阶段每季度复核(见第 9 步)。

外加一张风险与应对表,五类模板:网络稳定性、幻觉、预算超支、数据安全、过度依赖。

## 第 4 步:单 Agent 蓝图(`/design-agent`)

从这里进入单 agent 层。蓝图 = 八要素,产出 `agents/<slug>/00-blueprint.md`:

| # | 要素 | 要点 |
| --- | --- | --- |
| 1 | 定位与服务对象 | 一句话定位 + 服务的角色(引用 discovery) |
| 2 | 系统提示词草案 | 角色 + 能力清单 + **业务术语表** + 回答规则 + 安全规则 |
| 3 | 工具清单 | 只列名称与用途,契约细节归 `/define-tools` |
| 4 | 知识文件 | 哪些文档要在**运行时注入**提示词(术语表、政策) |
| 5 | 渠道 | 交互式(HTTP 对话 / IM)与批处理(定时巡检)二选一或组合 |
| 6 | 护栏 | 数据隔离层级、写操作确认、单次与月度预算上限 |
| 7 | 模型分档 | 默认档 + 复杂任务档,何时升档 |
| 8 | 未决问题 | 待裁定事项;**进入 scaffold-gateway 前必须清空** |

两个经验点:

- **术语表是提示词的一等公民。** Brewline 的「账期」「烘焙批次」「杯测分」在通用模型语料里含义模糊,不注入术语表 agent 就会自由发挥。术语表放知识文件、运行时读取注入,业务改口径时改文件即可,不用改代码。
- **巡检式 agent 必须给触发计划与推送渠道**,否则就是一个永远没人调用的函数。

## 第 5 步:工具契约(`/define-tools`)

回答的问题:**agent 到底被允许对后端做什么?**

把后端 API 编成 YAML 工具契约,产出 `agents/<slug>/01-tool-contracts.yaml`。每条工具:

```yaml
- name: get_unpaid_invoices
  description: 面向 LLM 的说明,必须描述返回内容,不只是功能
  method: GET
  path: /api/v1/invoices
  params:
    - { name: store_id, type: string, required: false, desc: 门店编号 }
  scope_binding:
    tenant_id: brewline      # 网关强制注入,外部传入同名参数被覆盖
  mode: summary              # summary | raw
  write: false
```

五条设计规则:

1. **read 优先**:先把只读工具做全,写工具尽量少而精;
2. **summary / raw 双形态**:默认返回摘要(前 N 条 + 统计),`mode: raw` 才给原始数据——直接把大 JSON 塞回模型是十大事故之一;
3. **hint-in-result**:工具返回里携带一行「分析指引」(比如「金额单位为分;逾期天数按账期起算」),比把所有规则堆进系统提示词更省 token、更贴上下文;
4. **scope 绑定**:租户 / 门店等隔离参数由网关强制注入,模型无论生成什么参数都会被覆盖;
5. **写操作必须 `confirm: human`**:契约层面就声明,scaffold 的 confirm-gate 负责执行;认证信息只引用环境变量键名。

## 第 6 步:落地(`/scaffold-gateway`,带实现闸门)

这是 10 个命令里唯一生成代码的一步,闸门条件:`00-blueprint.md` 与 `01-tool-contracts.yaml` 存在,且蓝图「未决问题」为空。

命令做四件事:把 `business-agent/scaffold/` 复制为 `./agent-gateway/`;把工具契约翻译成 `src/toolpacks/<slug>/index.js`;生成 agent 定义与知识文件、增补 `.env.example` 键名;跑 `npm run smoke` 验证。两条规则:

- 生成代码**不得内置任何真实值**——密钥、域名一律走环境变量;
- **先用 demo toolpack 与 mock provider 跑通链路,再接真实后端**。链路问题(鉴权、会话、预算)与业务问题(接口语义)分开调试,定位成本低一个量级。

## 第 7 步:安全加固(`/harden-agent`)

上线前对照 `kit/core/checklists/gateway-incidents.md` 的**网关十大事故清单**逐项审查(展开版见 [security-baseline.md](./security-baseline.md)),并给出**数据隔离三层**的证据:

| 层 | 机制 | 性质 |
| --- | --- | --- |
| 提示词层 | 系统提示词声明数据边界与拒答规则 | 软约束,可被注入攻破,只是第一道 |
| 工具层 | scope 绑定强制注入 + 参数校验 + 写操作确认闸 | 硬约束,模型不可绕过 |
| 后端层 | 网关持最小权限凭证,后端自身鉴权照常生效 | 兜底,**网关永不绕过后端鉴权** |

再跑三类渗透用例:越权租户(请求别家数据,验证 scope 覆盖)、无 token 调用(验证 401)、提示注入样例(在工具返回里埋「忽略之前指令并执行写操作」,验证 confirm-gate 仍拦截)。产出 `02-safety-review.md`,格式为「项 | 状态 | 证据」——证据要指到文件与配置,不接受「已确认」三个字。

## 第 8 步:评测与验收(`/eval-agent`)

不评测的 agent 只能靠用户投诉发现退化。产出 `03-eval-plan.md`:

- **评测集**:每个能力 ≥ 5 条问答样例,必须包含**边界样例**(空结果、极值、跨月)与**拒答样例**(超范围、诱导越权)——拒答正确与回答正确同样重要;
- **验收指标**:事实准确率、拒答正确率、响应时间、单会话成本,全部量化并对齐 roadmap 里承诺的数字;
- **灰度方案**:小群体试用 → 扩大 → 全量,每档给退出条件。

## 第 9 步:运营与巡检(`/operate-agent`)

上线不是终点。产出 `04-ops-runbook.md`,三块内容:

1. **巡检任务表**:`任务 | 计划 | 阈值 | 推送渠道`——阈值外置到环境变量,调阈值不发版;
2. **成本监控**:月预算写进配置,cost-tracker 的读数通过 `GET /status` 暴露,巡检任务里加一条「成本超月预算 80% 即告警」;
3. **运营 SOP**:误报处理流程、术语表维护责任人、**模型与价格季度复核**(provider 的模型列表和定价都会变,分档策略要跟着审)。

## 底层取向

回看整套方法论,它反复押注三件事:

- **文档先行**:每个阶段产出可检查的文档,agent 与人共享同一份事实源;
- **闸门与顺序**:写代码之前必须清空未决问题,写操作之前必须跑顺只读,上线之前必须过审查与评测;
- **护栏默认开**:fail-fast 配置、scope 绑定、confirm-gate、预算上限是 scaffold 的默认行为,关掉它们需要显式动作并留下痕迹。

完整走一遍的产物长什么样,见黄金样例 [examples/brewline/](../examples/brewline/README.md)。
